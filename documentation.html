CMS & Backend Runtime

ℹ️

Classic theme architecture

The site uses a WordPress Classic theme (not a block/FSE theme). This was a deliberate choice: the design calls for hand-authored, pixel-controlled templates — something block themes make harder without additional complexity. All markup lives in PHP template files with no Gutenberg block registration on the front end.

WordPress

6.x

CMS Platform

Open-source PHP content management system. Used as the application shell: it handles routing, the admin dashboard, plugin activation, user authentication, and the REST API. All content (pages, posts, settings) lives in the WP database.

PHP

≥ 8.0

Server-side language

All theme templates, plugin logic, and database interaction are written in PHP 8+. Typed parameters, return types, and match expressions are used throughout. PHP-CS-Fixer (via PHPCS + WordPress Coding Standards) enforces style in CI.

MySQL / MariaDB

via Local WP

Relational database

The standard WordPress database backend. The two custom plugins create their own dedicated tables (wp_swe_applications, wp_swe_waitlist) using dbDelta() — the WP-safe schema migration helper.

WordPress REST API

WP Core

Application API layer

Used by the ATS Form plugin (swe-fellows-application) to expose a JSON endpoint for AJAX form submission. The custom REST route validates and sanitizes every input server-side before writing to the database.

Frontend Stack

🎯

Zero build step, zero bundler

The frontend uses vanilla CSS and JavaScript with no preprocessor, no bundler, and no npm. The single stylesheet (assets/css/main.css) is a hand-authored token-driven file. JavaScript is split into two focused modules (nav.js, blocks.js) — both deferred and loaded from the footer. AOS is the only runtime library and is loaded from jsDelivr CDN (already mirrored in the browser cache for most users).

CSS (Custom Properties)

~1 750 lines

Design token system + component styles

All visual decisions are driven by a single token layer in :root — colors, type scale, spacing, and transitions. Components never hard-code values; they reference tokens. This makes a full palette swap a one-file change.

JavaScript (ES6+)

Vanilla, no framework

Navigation & scroll behavior

nav.js: mobile hamburger toggle with aria-expanded and keyboard trap. blocks.js: initialises AOS and wires any home-page block interactions. Both files are defer-loaded so they never block rendering.

AOS — Animate On Scroll

2.3.4

Scroll-triggered reveal animations

Lightweight (6 KB gzip) CSS animation library. Used for data-aos="fade-up" reveals on section cards and stat rows. Fully respects prefers-reduced-motion — animations are skipped when the OS motion setting is on.

HTML5 Semantics

WHATWG

Markup standard

All templates use semantic elements: <header>, <nav>, <main>, <article>, <aside>, <footer>, <details> / <summary> for the FAQ accordion (no JS needed), and native <table> for deadline data.

⚡

Performance rationale for self-hosting

Both fonts are served from the same origin as the page — eliminating the extra DNS lookup, TCP handshake, and font-fetch waterfall that Google Fonts incurs. They're <link rel="preload">-hinted in functions.php so the browser fetches them before the CSS parser discovers the @font-face rule. font-display: swap keeps text visible immediately.

Development Environment

Local WP

by WP Engine

Local development server

One-click local WordPress environment. Provides Apache/Nginx, PHP, and MySQL as isolated containers per site. Used throughout development — the site URL is https://northeastern-research.local. Xdebug is available via the bundled PHP build.

Git + GitHub

Version control

Source control & collaboration

Full project history is tracked in a GitHub repository on the custom-plugins-and-SCF base branch and a documentation feature branch. Conventional commit messages are used throughout (feat:, fix:, ci:, docs:).

GitHub Actions

CI/CD

Continuous integration pipeline

Automated linting runs on every push and pull request. The CI workflow installs PHP, Composer, and PHPCS with the WordPress Coding Standards ruleset, then runs phpcs over all theme and plugin PHP files. CSS and JS files are excluded; the project has no compiled assets to lint.

PHP CodeSniffer (PHPCS)

+ WPCS

Static analysis & style enforcement

phpcs.xml at the project root configures the WordPress and WordPress-Extra rulesets. It excludes the wp-admin/ and wp-includes/ core directories, vendor paths, and the generated CSS/JS. All custom code passes with zero errors.

CI Pipeline Configuration

The GitHub Actions workflow (.github/workflows/phpcs.yml) runs automatically on every push and pull request. It catches code-style and security-smelling patterns before they land on the main branch.

# .github/workflows/phpcs.yml (abbreviated)
name: PHP CodeSniffer

on: [push, pull_request]

jobs:
  phpcs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.1'
          tools: cs2pr

      - name: Install WPCS via Composer
        run: |
          composer global require dealerdirect/phpcodesniffer-composer-installer
          composer require --dev squizlabs/php_codesniffer
          composer require --dev wp-coding-standards/wpcs

      - name: Run PHPCS
        run: vendor/bin/phpcs --report=checkstyle | cs2pr
✅

Zero-error baseline

All custom theme and plugin PHP files pass the WordPress and WordPress-Extra rulesets with zero errors and zero warnings as of the initial commit. The phpcs.xml project config pins the ruleset, excludes WP core directories and vendor paths, and ignores CSS/JS files (which PHPCS cannot meaningfully lint).

External Services & Requests

The live site makes a carefully controlled set of external requests. The design principle is no unnecessary third-party origins — each one adds latency and a privacy surface.

Origin What loads Context Removable?
cdn.jsdelivr.net AOS v2.3.4 (CSS + JS) Front end, deferred Yes — replace with a local copy
None Fonts — self-hosted from theme origin Front end, preloaded N/A
None Analytics — Burst Statistics runs locally Admin + front end N/A

Google Fonts, Gravatar, and the default WordPress emoji CDN are all explicitly removed in functions.php via wp_dequeue_style(), remove_action(), and filter overrides.
