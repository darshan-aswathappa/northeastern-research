CMS & Backend Runtime

ℹ️

Classic theme architecture

The site uses a WordPress Classic theme (not a block/FSE theme). This was a deliberate choice: the design calls for hand-authored, pixel-controlled templates — something block themes make harder without additional complexity. All markup lives in PHP template files with no Gutenberg block registration on the front end.

WordPress

6.x

CMS Platform

Open-source PHP content management system. Used as the application shell: it handles routing, the admin dashboard, plugin activation, user authentication, and the REST API. All content (pages, posts, settings) lives in the WP database.

PHP

≥ 8.0

Server-side language

All theme templates, plugin logic, and database interaction are written in PHP 8+. Typed parameters, return types, and match expressions are used throughout. PHP-CS-Fixer (via PHPCS + WordPress Coding Standards) enforces style in CI.

MySQL / MariaDB

via Local WP

Relational database

The standard WordPress database backend. The two custom plugins create their own dedicated tables (wp_swe_applications, wp_swe_waitlist) using dbDelta() — the WP-safe schema migration helper.

WordPress REST API

WP Core

Application API layer

Used by the ATS Form plugin (swe-fellows-application) to expose a JSON endpoint for AJAX form submission. The custom REST route validates and sanitizes every input server-side before writing to the database.

Frontend Stack

🎯

Zero build step, zero bundler

The frontend uses vanilla CSS and JavaScript with no preprocessor, no bundler, and no npm. The single stylesheet (assets/css/main.css) is a hand-authored token-driven file. JavaScript is split into two focused modules (nav.js, blocks.js) — both deferred and loaded from the footer. AOS is the only runtime library and is loaded from jsDelivr CDN (already mirrored in the browser cache for most users).

CSS (Custom Properties)

~1 750 lines

Design token system + component styles

All visual decisions are driven by a single token layer in :root — colors, type scale, spacing, and transitions. Components never hard-code values; they reference tokens. This makes a full palette swap a one-file change.

JavaScript (ES6+)

Vanilla, no framework

Navigation & scroll behavior

nav.js: mobile hamburger toggle with aria-expanded and keyboard trap. blocks.js: initialises AOS and wires any home-page block interactions. Both files are defer-loaded so they never block rendering.

2.3.4

Scroll-triggered reveal animations

Lightweight (6 KB gzip) CSS animation library. Used for data-aos="fade-up" reveals on section cards and stat rows. Fully respects prefers-reduced-motion — animations are skipped when the OS motion setting is on.

⚡

Performance rationale for self-hosting

Both fonts are served from the same origin as the page — eliminating the extra DNS lookup, TCP handshake, and font-fetch waterfall that Google Fonts incurs. They're <link rel="preload">-hinted in functions.php so the browser fetches them before the CSS parser discovers the @font-face rule. font-display: swap keeps text visible immediately.

Theme Overview

🎯

Why a Classic theme, not a Block/FSE theme?

The design system calls for pixel-precise editorial control over every section of every page — the hero billboard, the stat rows, the timeline, the FAQs. A block/FSE theme would require registering patterns and templates in the block editor, adding complexity without benefit. Classic PHP templates give direct, readable ownership of every byte of markup; the block system is still used for the home-page content, but via server-rendered callbacks rather than client-rendered blocks.

functions.php Architecture

All theme logic lives in a single functions.php. It is structured as a flat sequence of named functions, each bound to a WordPress hook — no classes, no namespaces, no autoloader. This keeps the file scannable and matches the WP Coding Standards for classic themes.

Theme Setup

nu_research_setup()

after_setup_theme

Declares theme supports: title-tag, post-thumbnails, responsive-embeds, html5 (all subtypes), and editor styles. Registers the single primary nav menu location.

Asset Enqueuing

nu_research_enqueue()

wp_enqueue_scripts

Enqueues main.css with a filemtime() version (cache-busting on every deploy). Also enqueues AOS CSS + JS from jsDelivr CDN, and nav.js — both with strategy: defer so they never block rendering.

nu_research_dequeue_unused()

wp_enqueue_scripts (p.100)

Removes four WP-generated stylesheets that classic themes don't need: wp-block-library, wp-block-library-theme, classic-theme-styles, and global-styles. Runs at priority 100 to ensure it fires after any plugin registration.

nu_research_disable_extras()

init

Strips five more sources of front-end weight: the emoji detection script, emoji CSS, the oEmbed host JS, the global-styles generating actions (two separate locations), and the global SVG filters for block styles. Keeps all of them in the admin so the editor is unaffected.

Performance — Head Preloads

nu_research_preloads()

wp_head (p.1)

Emits <link rel="preload"> for both variable fonts (so text renders before the CSS parser discovers @font-face), the hero background image on the front page (the LCP element — CSS backgrounds are invisible to the preload scanner without this hint), and the full-bleed photo on the About page.

SEO Fallback

nu_research_meta_description()

wp_head (p.1)

Emits a <meta name="description"> using the page excerpt (if set) or the site tagline. Yields silently when Yoast SEO (WPSEO_VERSION constant) is active — Yoast owns the meta description, Open Graph, and canonical, so the theme never duplicates those tags.

JavaScript Modules

nav.js

~32 lines

Front end · deferred · IIFE

Accessible mobile-first hamburger disclosure. Adds .nav-ready to <html> (the CSS collapsed state only activates once this class is present, so no-JS users always see the nav). Manages aria-expanded on the toggle and data-collapsed on the nav. Handles Escape-key close (with focus return to the toggle) and a matchMedia listener that resets state on viewport resize.

blocks.js

Editor only

Admin · enqueued on enqueue_block_editor_assets

Registers editor UIs for all eight custom blocks. Uses a data-driven BLOCKS array so adding a new block requires only a new object entry — no boilerplate duplication. Each block gets InspectorControls with typed inputs (text, textarea, toggle, select, lines) and a live ServerSideRender preview. No JSX, no Webpack.

After AOS is loaded, nav.js also initialises it on the window load event: 

AOS.init({
  duration: 600,  // ms per animation
  once:     true, // animate only on first entry
  offset:   80,   // px from viewport edge
});

prefers-reduced-motion is handled at the CSS level — the theme's global reset sets all transition/animation durations to 0.01ms when the OS motion preference is on, which effectively cancels AOS animations without any extra JS.

Fellows Deadline Countdown

Slug fellows-deadline
Version 1.1.0

Purpose

Central date-management plugin for the entire application lifecycle. It stores three dates in wp_options and exposes them to every other part of the system — the theme, the ATS Form plugin, and the Mail List plugin all read from these same option keys to stay in sync.

🔑

Single source of truth for dates

The three option keys (fellows_dl_open, fellows_dl_close, fellows_dl_notify) are the authoritative source for whether applications are open. The ATS Form plugin, the Mail List waitlist plugin, the utility-bar pill, and the closing-soon banner all derive their state from these values — no duplication, no drift. Once the window closes, fellows_dl_notify drives a "Decisions expected …" state across the badge and pill.

Features

Settings Page

Settings → Fellows Deadline

Three date fields (open, close, notify) registered via the WordPress Settings API with Y-m-d sanitisation. Inputs are validated with DateTime::createFromFormat() — invalid dates are silently cleared. A separate "Closing-Soon Banner" section holds the banner lead time (fellows_dl_banner_days), a whole number of days clamped to 1–90.

[fellows_countdown] shortcode

Front end

Renders a status display — "X days until applications close" or "Applications open Dec 1". Output adapts to the three date states: pre-open, open, or closed. Accepts two attributes — show_days and show_notify (both default true); once closed it surfaces the decision date as "Decisions expected …". Styled by assets/deadline.css.

Security Measures

Honeypot field

A hidden swe_website field that real users never fill. If it contains any value the submission is rejected with a success-shaped 400 (so bots learn nothing useful). Checked in swe_app_validate() before any other processing.

Per-IP rate limiting

Maximum 5 submissions per 10-minute window per IP address. Enforced via WordPress transients (swe_app_rl_{md5(ip)}). Returns HTTP 429 on breach. Checked via swe_app_check_rate_limit() after validation, so honest mistakes and bot noise don't consume the quota.

Server-side allowlists

Class year and track values are validated against PHP arrays — not just sanitised. A crafted request with an arbitrary track value is rejected. Allowlists are defined in swe_app_class_years() and swe_app_tracks().

Prepared statements

All database writes use $wpdb->insert() with format specifiers (%s, %d). All reads that include user-supplied values use $wpdb->prepare(). No string-built SQL anywhere in the plugin.

Resume upload hardening

Uploaded resumes are validated as genuine PDFs (extension + MIME sniffing via wp_check_filetype_and_ext() and a %PDF- magic-byte check), capped at 5 MB, and only moved to disk after every other check passes. Files are stored under uploads/swe-resumes/ with unguessable randomized names; an index.html guard blocks directory listing, and deleting an application removes its resume file.

Design Rationale

🗄️

Why dedicated tables instead of posts/postmeta?

WordPress's generic post system (wp_posts + wp_postmeta) is designed for content, not structured data with typed columns, queryable fields, and status workflows. Storing applications in wp_posts would mean no type safety, no indexed track or email columns, no clean SQL GROUP BY for the dashboard widget, and a postmeta row per field instead of a single typed row. Dedicated tables give plain SQL access, proper column types, indexes on the columns actually queried, and a schema that documents intent clearly.

⬆️

Migrations via dbDelta()

dbDelta() is WordPress's built-in schema migration utility. It compares the desired CREATE TABLE statement against the live schema and applies only the differences — adding new columns, adding indexes — without dropping existing data. Both plugins call it on activation via register_activation_hook and on every plugins_loaded via an upgrade-version check, so schema changes are applied even when a plugin is updated without deactivation.

Duplicate Handling & Race Condition

⚡

Two-layer duplicate prevention

swe_ml_subscribe() first calls swe_ml_email_exists() (an application-level check), then issues the $wpdb->insert(). If two concurrent requests both pass the application check before either writes to the database, the second insert will fail at the MySQL UNIQUE KEY constraint. The catch block then calls swe_ml_email_exists() again — if the email now exists, it returns a "duplicate" error rather than a generic server error. This covers the TOCTOU (time-of-check/time-of-use) race without needing a transaction.

SQL Patterns & Safety

All database interaction in the custom plugins follows the same safety rules — no exceptions.

Rule Implementation
No raw SQL with user input Every query that includes a user-supplied or external value uses $wpdb->prepare() with %s / %d placeholders. No string concatenation for WHERE clauses.
Writes use $wpdb->insert() / update() / delete() These methods accept a data array and a format array separately — the driver handles escaping. No hand-built INSERT or UPDATE strings anywhere in the codebase.
Table names are never user-supplied swe_app_table_name() and swe_ml_table_name() construct the name from $wpdb->prefix (a trusted config value) plus a static string. PHPCS flags interpolated table names; the // phpcs:ignore suppression is only used on lines where the table name is provably safe.
Input sanitised before storage sanitize_text_field(), sanitize_email(), sanitize_textarea_field() are applied in the validation layer (swe_app_validate(), swe_ml_subscribe()) before the data reaches any database function.
Allowlist validation on enums class_year, track, and status are checked against PHP arrays (in_array(..., true) strict mode) before insert or update. A crafted request with an out-of-range value is rejected at the validation layer, never at the DB layer.

Design Philosophy

🎯

No gradients (the hero photo scrim is the one documented exception). No box shadows. No rounded corners beyond the pill filter-bar, which is the system's single sanctioned rounded element.

Flat surfaces

No card shadows, no elevation levels, no depth cues. Sections are separated by hairline 1 px borders (var(--color-border): #e6e6e6) and background-fill contrast (#f4f4f4 muted vs. #ffffff white). The UI reads as editorial print, not a web app.

Square corners everywhere

border-radius: 0 on all buttons, inputs, cards, and image containers. The only exception is the category filter-pill bar, defined by --radius-pill: 999px and restricted to that single component.

Density as a virtue

The audience is graduate-program applicants and faculty — technically literate, goal-oriented. Layouts prefer information density over whitespace breathing room. Section padding scales from 48 px on mobile to 96 px on large screens, but within sections content is compact.

Hero Billboard — Media Bleed

The home-page billboard hero achieves a photo panel that bleeds to the browser edge on large screens while the copy stays within the .wrap content boundary. This is done with negative margins: 

/* Mobile: photo bleeds to both sides (full-width) */
.hero-billboard-media {
  margin-inline: calc(-1 * var(--content-pad));
  margin-bottom: calc(-1 * clamp(40px, 6vw, 96px));
}

/* ≥ 900px: photo bleeds only to the right edge */
@media (min-width: 900px) {
  .hero-billboard-media {
    margin-inline: 0;
    margin-top: calc(-1 * clamp(40px, 6vw, 96px));
    margin-bottom: calc(-1 * clamp(40px, 6vw, 96px));
    /* right bleed = content-pad + half the gap between wrap and viewport edge */
    margin-right: calc(-1 * (var(--content-pad) + max(0px, (100vw - var(--content-max-width)) / 2)));
  }
}

The max(0px, ...) guard ensures the right margin never goes positive (which would push the photo inward) on viewports narrower than --content-max-width.

A global :focus-visible rule provides a consistent 2 px red outline (outline: 2px solid var(--nu-red); outline-offset: 2px) on all interactive elements when navigated by keyboard. :focus-visible — not :focus — is used so mouse clicks do not show the outline (matching browser native behaviour for styled elements).

This ensures the focus indicator is always visible against the surface it sits on — a contrast requirement, not just an aesthetic one.

Reduced Motion

The theme respects the OS-level "Reduce Motion" preference via the prefers-reduced-motion: reduce media query. The global reset in main.css sets all animation and transition durations to 0.01 ms when the preference is active: 

@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

This single rule cancels all AOS scroll-triggered reveals, all CSS button/link hover transitions, and all CSS FAQ accordion transitions — without needing to find and override each animation individually. The !important is intentional and correct here: it must override any inline or specificity-winning animation declarations.

Database Security

Prepared statements

Every query that includes a variable uses $wpdb->prepare() with typed placeholders (%s, %d). The driver handles escaping and quoting — no string concatenation for WHERE clauses anywhere in the codebase.

Structured write methods

$wpdb->insert(), $wpdb->update(), and $wpdb->delete() accept a data array and a format array separately. The driver handles all quoting. No hand-built INSERT or UPDATE strings.

Server-side allowlists

Enum-like columns (class_year, track, status) are validated against strict PHP arrays before any database operation. A crafted request with an out-of-range value is rejected at the validation layer — never at the DB layer.

UNIQUE constraint on emails

The wp_swe_waitlist.email column has a MySQL UNIQUE KEY. Even if application-level duplicate checking is bypassed (race condition), the database rejects duplicate inserts at the engine level.

CSRF Protection & Nonces

All state-changing form submissions are protected by WordPress nonces — short-lived signed tokens that verify the request originated from the legitimate page and the current authenticated session.

Spam & Bot Protection

Honeypot fields

Both the application form (swe_website) and the waitlist form include a hidden field that real users never fill. If the field contains any value, the submission is rejected with a success-shaped response — bots learn nothing useful. Checked before any other validation.

Per-IP rate limiting

Both submission endpoints limit to 5 requests per 10 minutes per IP address using WordPress transients (set_transient($key, $count, 10 * MINUTE_IN_SECONDS)). Breaches return HTTP 429. Rate-limit keys use md5($ip) to avoid storing raw IPs.

WAF (All-in-One Security)

The All-in-One WP Security plugin writes firewall rules to .htaccess blocking common XSS probes, SQL injection fragments, directory traversal attempts, and suspicious user-agent strings before they reach PHP.

Login URL obfuscation

WPS Hide Login moves wp-login.php to a custom slug. Automated scanners targeting the default path receive a 404. This eliminates the overwhelming majority of credential-stuffing bots, which do not enumerate login URLs.

Output & XSS Prevention

The WordPress Coding Standard enforced by PHPCS in CI requires every template echo to use a contextually appropriate escape function. The CI pipeline fails on any unescaped output — this is enforced at the code level, not just by convention.

Direct File Access Prevention

Every custom plugin file opens with the WordPress direct-access guard: 

if ( ! defined( 'ABSPATH' ) ) {
    exit;
}

ABSPATH is only defined when WordPress bootstraps. If a file is accessed directly via HTTP (e.g., https://example.com/wp-content/plugins/fellows-deadline/fellows-deadline.php), the constant is undefined and the script exits immediately — no PHP execution, no error messages that could reveal server paths or PHP version.

Static Analysis in CI

PHP CodeSniffer with the WordPress and WordPress-Extra rulesets runs on every push via GitHub Actions. The WordPress ruleset flags:

  • Missing output-escaping functions (esc_html, esc_attr, esc_url)
  • Unescaped SQL (direct database queries without prepare())
  • Missing nonce verification on form handlers
  • Missing capability checks on admin callbacks
  • Use of deprecated / unsafe PHP functions
  • Direct superglobal access without nonce verification ($_POST, $_GET)

All custom code passes with zero errors and zero warnings. The phpcs.xml project config documents every intentional suppression inline with a justification comment.
